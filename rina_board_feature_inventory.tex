\section{Hardware, matrix, and board features}
\subsection{LED hardware}
\begin{itemize}
  \item LED data pin: \textbf{GPIO 15}
  \item Total LEDs: \textbf{370}
  \item Driver type: \textbf{NeoPixel / WS2812B}
  \item Runtime target FPS helper in \texttt{board.py}: \textbf{60 FPS}
  \item Frame time: \texttt{FRAME\_TIME\_S = 1/60}, \texttt{FRAME\_TIME\_MS = 1000/60}
\end{itemize}

\subsection{Matrix geometry}
The logical display is treated as \textbf{22 columns $\times$ 18 rows}, but with irregular row lengths.

\begin{lstlisting}
Row lengths:
18,
20, 20, 20,
22, 22, 22, 22, 22, 22, 22, 22, 22,
20, 20, 20,
18,
16
\end{lstlisting}

Therefore:
\begin{itemize}
  \item \texttt{ROWS = 18}
  \item \texttt{COLS = 22}
\end{itemize}

\subsection{Wiring and orientation logic}
\begin{itemize}
  \item \texttt{SERPENTINE = True}
  \item \texttt{FLIP\_X = False}
  \item \texttt{FLIP\_Y = False}
\end{itemize}

Meaning:
\begin{itemize}
  \item Odd rows are reversed in physical LED order.
  \item No X flip is applied.
  \item No Y flip is applied.
\end{itemize}

\subsection{Board drawing helpers}
The board layer provides:
\begin{itemize}
  \item brightness limiting
  \item logical \texttt{(x, y)} to physical LED index mapping
  \item clear, fill, and show helpers
  \item bitmap drawing
  \item bitmap blending
  \item pixel-grid drawing
  \item wheel and HSV color helpers
  \item gamma helper
  \item radial falloff helper
  \item linear interpolation and blending helpers
\end{itemize}

\section{Brightness system}
\subsection{User-facing brightness values}
Stored brightness is a percentage rather than a raw LED value.

\begin{center}
\begin{tabular}{ll}
\toprule
Constant & Value \\
\midrule
\texttt{DEFAULT\_BRIGHTNESS} & 30 \\
\texttt{BRIGHTNESS\_STEP} & 5 \\
\texttt{BRIGHTNESS\_MIN} & 5 \\
\texttt{BRIGHTNESS\_MAX} & 100 \\
\texttt{BRIGHTNESS\_MAX\_CHANNEL} & 170 \\
\bottomrule
\end{tabular}
\end{center}

So the user brightness range is \textbf{5\% to 100\%}, in \textbf{5\% steps}, with default \textbf{30\%}.

\subsection{Physical LED cap logic}
The code maps UI percent to a raw per-channel cap using:
\begin{lstlisting}
cap = (ui_brightness * 170 + 50) // 100
\end{lstlisting}

Examples:
\begin{itemize}
  \item 100\% $\rightarrow$ 170
  \item 50\% $\rightarrow$ 85
  \item 30\% $\rightarrow$ 51
  \item 5\% $\rightarrow$ 8
\end{itemize}

\subsection{Mode-based brightness scaling}
Actual effective cap:
\begin{itemize}
  \item \textbf{face mode}: full mapped cap
  \item \textbf{Bad Apple mode}: cap divided by \textbf{3}
  \item \textbf{demo mode}: cap divided by \textbf{3}
\end{itemize}

Set values:
\begin{itemize}
  \item \texttt{BADAPPLE\_BRIGHTNESS\_DIVISOR = 3}
  \item \texttt{DEMO\_BRIGHTNESS\_DIVISOR = 3}
\end{itemize}

Important detail: some comments and README text say Bad Apple is ``half brightness,'' but the actual code uses \textbf{one-third}, not one-half.

\subsection{Brightness reset}
Default brightness reset goes to \textbf{30\%}.

Reset combo logic:
\begin{itemize}
  \item hold \textbf{Button 4 + Button 5 together}
  \item guarded by a latch so it only fires once until both buttons are released
  \item ignore window after firing: \texttt{BRIGHTNESS\_RESET\_IGNORE\_MS = 300}
\end{itemize}

\section{Battery measurement and battery display features}
\subsection{ADC hardware}
Battery ADC:
\begin{itemize}
  \item GPIO 26
  \item ADC reference 3.3 V
  \item 16 samples per instantaneous read
  \item divider: \texttt{BATTERY\_DIVIDER\_R1 = 100000}, \texttt{BATTERY\_DIVIDER\_R2 = 57000}
\end{itemize}

Charger detect ADC:
\begin{itemize}
  \item GPIO 27
  \item ADC reference 3.3 V
  \item 16 samples per instantaneous read
  \item divider: \texttt{CHARGE\_DETECT\_DIVIDER\_R1 = 270000}, \texttt{CHARGE\_DETECT\_DIVIDER\_R2 = 47000}
\end{itemize}

\subsection{Mean-sampling logic}
\begin{itemize}
  \item \texttt{BATTERY\_MEAN\_UPDATE\_MS = 1000}
  \item \texttt{BATTERY\_MEAN\_SAMPLE\_INTERVAL\_MS = 20}
\end{itemize}

So a new mean window completes every 1 second, with samples every 20 ms.

\subsection{Battery display refresh timing}
\begin{itemize}
  \item \texttt{BATTERY\_REFRESH\_MS = 100}
  \item \texttt{BATTERY\_ANIMATION\_REFRESH\_MS = 50}
  \item \texttt{BATTERY\_DISPLAY\_CYCLE\_MS = 2000}
\end{itemize}

Meaning:
\begin{itemize}
  \item battery cache refresh every 100 ms
  \item animation visual refresh every 50 ms
  \item each battery page stays on screen for 2 s
\end{itemize}

\subsection{Battery short display}
Tap Button 6:
\begin{itemize}
  \item activates a single-shot display
  \item lasts 2000 ms
  \item shows percent phase only
  \item still checks charge state every poll tick so unplug and replug updates immediately
\end{itemize}

\subsection{Battery long display}
Hold Button 6 for 700 ms:
\begin{itemize}
  \item enters full battery overlay cycling
  \item phase count depends on charging state
\end{itemize}

When not charging, the 3 pages are:
\begin{enumerate}
  \item battery percent
  \item battery voltage
  \item remaining time estimate
\end{enumerate}

When charging, the 4 pages are:
\begin{enumerate}
  \item battery percent
  \item battery voltage
  \item estimated charge time
  \item charger voltage
\end{enumerate}

\subsection{Immediate charge-state response}
The code intentionally reads the charge ADC directly every poll tick for fast unplug and replug detection, instead of waiting for the 1-second average.

\section{Battery percentage logic}
\subsection{Default calibration range}
\begin{itemize}
  \item \texttt{BATTERY\_DEFAULT\_MIN\_V = 6.2}
  \item \texttt{BATTERY\_DEFAULT\_MAX\_V = 8.0}
  \item \texttt{BATTERY\_DISPLAY\_TOL\_V = 0.12}
  \item \texttt{BATTERY\_MIN\_SPAN\_V = 0.20}
  \item \texttt{BATTERY\_CAL\_VERSION = 4}
\end{itemize}

\subsection{Percent curve}
Battery percent is not linear. It uses this lookup curve:
\begin{lstlisting}
(0.000,   0.0)
(0.222,   3.0)
(0.389,   7.0)
(0.444,  10.0)
(0.500,  14.0)
(0.556,  18.0)
(0.611,  26.0)
(0.667,  35.0)
(0.722,  45.0)
(0.778,  58.0)
(0.833,  70.0)
(0.889,  82.0)
(0.944,  92.0)
(1.000, 100.0)
\end{lstlisting}

Logic:
\begin{itemize}
  \item below \texttt{min\_v + 0.12} $\rightarrow$ 0\%
  \item above \texttt{max\_v - 0.12} $\rightarrow$ 100\%
  \item otherwise interpolate along the curve
\end{itemize}
Rounded integer percent is used for display.

\section{Battery color logic}
Battery percent color is based on percent:
\begin{itemize}
  \item $\leq 10\%$ $\rightarrow$ red \texttt{(255, 0, 0)}
  \item 10\% to 30\% $\rightarrow$ red to orange interpolation
  \item 30\% to 50\% $\rightarrow$ orange/yellow to green interpolation
  \item $> 50\%$ $\rightarrow$ green \texttt{(0, 255, 0)}
\end{itemize}

So the battery icon becomes red at very low percentage, orange or yellow in the low-mid range, and green from 50\% upward.

\section{Charging detection logic}
Actual charge-state logic in code:
\begin{itemize}
  \item if charge voltage $> 4.0$ V $\rightarrow$ charging
  \item if charge voltage $\leq 3.0$ V $\rightarrow$ not charging
  \item between 3.0 V and 4.0 V $\rightarrow$ keep previous state
\end{itemize}

Set values used:
\begin{itemize}
  \item \texttt{CHARGE\_DETECT\_CHARGING\_MIN\_V = 4.0}
  \item \texttt{CHARGE\_DETECT\_HYSTERESIS\_LOW\_V = 3.0}
\end{itemize}

This is a hysteresis and hold-state scheme to reduce flicker.

Important detail: these config values exist but are not used in current logic:
\begin{itemize}
  \item \texttt{CHARGE\_DETECT\_NON\_CHARGING\_V = 3.0}
  \item \texttt{CHARGE\_DISPLAY\_THRESHOLD\_V = 4.5}
\end{itemize}

The code uses 4.0 and 3.0, not 4.5.

\section{Battery animation logic}
\subsection{Fill logic}
Battery icon inner fill has \texttt{inner\_cols = 8}.

Conversion:
\begin{itemize}
  \item below 10\% $\rightarrow$ 0 filled columns
  \item above 90\% $\rightarrow$ all 8 columns
  \item otherwise percent maps to 1 through 8 columns
\end{itemize}

\subsection{Charging animation}
Set values:
\begin{itemize}
  \item \texttt{BATTERY\_CHARGE\_ANIM\_INTERVAL\_EMPTY\_S = 0.2}
  \item \texttt{BATTERY\_CHARGE\_ANIM\_INTERVAL\_NEAR\_FULL\_S = 0.2}
  \item \texttt{BATTERY\_CHARGE\_ANIM\_NEAR\_FULL\_PERCENT = 90}
  \item \texttt{BATTERY\_CHARGE\_ANIM\_FULL\_CYCLE\_S = 0.2}
  \item \texttt{BATTERY\_CHARGE\_LAST\_COLUMN\_FLASH\_MS = 300}
\end{itemize}

Actual logic currently used:
\begin{itemize}
  \item \texttt{charge\_animation\_step\_interval\_s(percent)} always returns 0.2 s
  \item below 10\% while charging: one column blinks on and off
  \item 10\% to 90\%: solid sweep from 1 column up to target filled columns, then loops
  \item above 90\%: sweeps across all 8 columns
  \item if not charging: battery shows static filled columns
\end{itemize}

Important detail: the ``near full'' constants exist, but current code does not vary the speed with percent. It always uses 0.2 s.

\section{Battery logging, calibration, and learned min-max}
\subsection{Logging interval}
\texttt{BATTERY\_LOG\_INTERVAL\_MS = 30000}, so battery history and calibration updates happen every 30 s.

\subsection{Discharge relearn settings}
\begin{itemize}
  \item \texttt{BATTERY\_RELEARN\_EVERY\_MEASUREMENTS = 2000}
  \item \texttt{BATTERY\_RELEARN\_MAX\_STEP\_V = 0.05}
  \item \texttt{BATTERY\_RELEARN\_MIN\_STEP\_V = 0.05}
  \item \texttt{BATTERY\_RELEARN\_HOLDOFF\_MEASUREMENTS = 20}
  \item \texttt{BATTERY\_RELEARN\_MAX\_CONSECUTIVE = 2}
\end{itemize}

Actual code behavior:
\begin{itemize}
  \item every 2000 discharge measurements, min increases by 0.05 V and max decreases by 0.05 V
  \item after inward adjustment, holdoff counter is set to 20
  \item while holdoff $> 0$, no outward relearning from live data occurs
  \item when holdoff expires, if observed voltage is below min then min updates down, and if observed voltage is above max then max updates up
\end{itemize}

Important detail:
\begin{itemize}
  \item \texttt{BATTERY\_RELEARN\_MAX\_CONSECUTIVE = 2} exists
  \item \texttt{inward\_min\_count} and \texttt{inward\_max\_count} exist and are persisted
  \item but that consecutive-limit logic is not actually enforced in current \texttt{battery\_monitor.py}
\end{itemize}

So those counters are currently dead state.

\subsection{Measurement count behavior}
\texttt{measure\_count} increments only when \textbf{not charging}.

\end{itemize}

\begin{itemize}

  \item \texttt{charge\_history\_last\_percent}
\end{itemize}

\subsection{History tuning constants}
\begin{itemize}
  \item \texttt{BATTERY\_HISTORY\_MAX\_SAMPLES = 96}
  \item \texttt{BATTERY\_HISTORY\_MIN\_RATE\_PCT\_PER\_H = 0.25}
  \item \texttt{BATTERY\_HISTORY\_SAME\_MODE\_WEIGHT = 2.5}
  \item \texttt{BATTERY\_HISTORY\_BRIGHTNESS\_WINDOW = 20}
  \item \texttt{BATTERY\_DEFAULT\_USAGE\_HOURS = 1.0}
  \item \texttt{BATTERY\_DEFAULT\_CHARGE\_HOURS = 0.5}
\end{itemize}

\subsection{Mode-aware estimation}
History entries are weighted by brightness similarity and mode similarity. Same mode gets extra weight \textbf{2.5}, and the brightness proximity window is \textbf{20}.

Mode code depends on state:
\begin{itemize}
  \item face mode
  \item demo mode
  \item badapple mode
\end{itemize}

\subsection{Fallbacks}
If there is no useful history:
\begin{itemize}
  \item default runtime estimate: 1.0 hour
  \item default charge estimate: 0.5 hour = 30 min
\end{itemize}

\subsection{Logging behavior by charging state}
\begin{itemize}
  \item when charging, records charge samples and does not add discharge samples
  \item when not charging, records discharge samples and does not add charge samples
\end{itemize}

\subsection{Included demos}

  \item step timing about 95 ms

\subsubsection*{Hello World demo}

  \item scrolls the text \texttt{" HELLO WORLD "}
  \item one full crossing over exactly the current demo interval
  \item text is vertically centered
  \item rainbow text coloring via color wheel
  \item if too few pixels are lit near edges, a tiny leading dot is shown

  \item brightness works in demo mode

\section{Bad Apple mode features}
\subsection{Mode entry and exit}
Combo: hold \textbf{Button 2 + Button 6} for 2 s.

Behavior:
\begin{itemize}
  \item entering Bad Apple first exits demo mode if needed
  \item brightness is reduced by divisor 3
  \item player loads segmented frame data
  \item on failure, mode exits and face display returns
\end{itemize}

\subsection{Playback data}
Segment modules loaded:
\texttt{badapple\_part0} through \texttt{badapple\_part16}. That means there are \textbf{17 parts}.

\subsection{Playback engine behavior}
\begin{itemize}
  \item split-part player
  \item non-looping: \texttt{loop=False}
  \item playback is non-blocking
  \item after overlays, \texttt{resync()} realigns frame timing to current time
  \item when playback reaches the end, it automatically exits Bad Apple mode and returns to normal face display
\end{itemize}

\subsection{Drawing logic}
\begin{itemize}
  \item frames are 1-bit packed bitmaps
  \item decoded and centered inside the 22 $\times$ 18 matrix
  \item colors: on = \texttt{(255,255,255)}, off = \texttt{(0,0,0)}
\end{itemize}

\subsection{Controls while active}
Allowed:
\begin{itemize}
  \item brightness up and down
  \item brightness reset display logic
  \item battery short and long display
  \item Button 2 + Button 6 exit combo
\end{itemize}

Blocked:
\begin{itemize}
  \item face cycling
  \item auto toggle
  \item interval controls
  \item demo controls
\end{itemize}

\section{Auto-timer logic}
\subsection{Face mode auto-cycle}
If \texttt{auto=True} and no overlay is active, the face increments when \texttt{next\_auto\_ms} is reached, using \texttt{state.interval\_s}.

\subsection{Timer reset cases}
The auto timer is reset when:
\begin{itemize}
  \item any button press is handled
  \item flash overlay is active
  \item battery display is active
  \item a combo is active
\end{itemize}

So overlays and combos prevent accidental auto-advance during interaction.

\section{Startup behavior}
On boot, the firmware:
\begin{enumerate}
  \item prints startup info
  \item loads saved settings
  \item prints learned min and max
  \item applies brightness
  \item draws current face
  \item applies demo runtime settings
  \item takes an initial battery sample
  \item forces calibration and log update once
  \item enters the infinite polling loop
\end{enumerate}

Startup info printed includes:
\begin{itemize}
  \item default face interval
  \item default demo interval
  \item default brightness
  \item button GPIO list
  \item battery ADC GPIO
  \item battery divider text
  \item demo combo text
  \item Bad Apple combo text
\end{itemize}

\section{Main loop logic}
Every loop:
\begin{enumerate}
  \item checks Button 3 + Button 6 combo
  \item checks Button 2 + Button 6 combo
  \item polls buttons
  \item handles button events
  \item rechecks combos
  \item checks Button 6 hold behavior
  \item services battery overlay
  \item clears brightness-reset latch if Buttons 4 and 5 are released
  \item checks Button 3 release
  \item checks Button 6 release
  \item expires flash if needed
  \item draws edge flash if needed
  \item does background battery sampling and calibration if allowed
  \item runs Bad Apple if active
  \item else runs demo mode if active
  \item else runs normal face auto-cycle if enabled
  \item sleeps for \texttt{POLL\_PERIOD\_MS = 10}
\end{enumerate}

\subsection{Background battery sampling rule}
\begin{lstlisting}
allow_background_sampling = (not badapple_mode) or battery_display_active
\end{lstlisting}

Meaning:
\begin{itemize}
  \item during Bad Apple, background sampling is mostly paused
  \item but battery sampling is allowed while battery display is active
\end{itemize}

\subsection{Comment and README mismatches}
\begin{itemize}
  \item Some comments say Bad Apple uses half brightness, but actual code uses divisor 3.
  \item README mentions relearn every 1000 discharge measurements, but actual code uses \texttt{BATTERY\_RELEARN\_EVERY\_MEASUREMENTS = 2000}.
  \item README mentions a charge threshold of 4.5 V, but actual code uses charging if above 4.0 V and not charging if at or below 3.0 V.
\end{itemize}

So the code and README are not fully in sync.
